## 요약

2026년 3월 31일 Anthropic Claude Code의 소스맵 파일이 npm 배포 사고로 유출된 사건을 계기로, 에이전트 완성도가 어디서 갈리는지를 분석한 글. 에이전트는 모델 자체보다 "어떤 루프와 상태 위에서 움직이는가"에서 차이가 난다는 주장을 중심으로 한다.

## 에이전트의 정의

- 중세 라틴어 agens/agent-: "움직이게 하다, 이끌다, 행하다"
- Russell & Norvig (AIMA 초판 1995): "환경을 센서로 지각하고 액추에이터로 작용하는 것"
- **현재**: "어떤 루프와 상태 위에서 움직이는가"가 핵심

### 빅테크의 에이전트 정의 비교

| 요소 | 설명 |
|------|------|
| **Runtime Loop** | 도구 호출 → 결과 수신 → 다음 행동의 반복 구조 |
| **Orchestration** | 전체 흐름 조율 레이어 |
| **Handoffs** | 다른 에이전트나 사람에게 작업 이전 |
| **Guardrails** | 위험한 행동 차단/제한 |
| **State** | 작업 진행 상태와 맥락 유지 |
| **Observability** | 로그, 메트릭, 트레이스로 시스템 행동 추적 |

## Claude Code 유출에서 배운 것

### 1. Agent의 핵심은 Agent Loop

유출된 실제 흐름:
1. 모든 대화가 단 하나의 함수 `submitMessage()` 통과
2. System Prompt 결합
3. 내부적으로 최소 **42개 도구** 동적 배치
4. 컨텍스트 과다 시 → 이전 메시지 자동 요약·압축
5. 프로젝트 → 사용자 → 글로벌 메모리 **계층적 로딩**

```python
# Agent loop 핵심 구조 (의사코드)
while turn < max_turns:
    runtime_context = build_runtime_context(messages, memory, task_state)
    llm_response = model.generate(runtime_context, available_tools=tools)
    if llm_response.type == "final": return llm_response.text
    if llm_response.type == "tool_use":
        decision = policy.check(call)  # allow / ask / deny / escalate
        # 실행 → 결과 메시지에 재주입
```

### 2. 툴 계층 설계: 위험도별 분리

```python
# 위험도별 툴 정책
read-only     → 즉시 실행
external-read → 즉시 실행 (외부 접근)
write         → Human Review 필요
exec          → Human Review 필요
destructive   → 차단
```

### 3. "만능 한 명" → "역할이 나뉜 팀 구조"

```
manager → diff 요약 → review specialist → 승인 후 수정 agent
```

- 각 단계의 [입력, 출력, 권한]이 좁고 분명해야 함
- Anthropic 공식: 각 subagent는 자기 context window + 별도 system prompt + 특정 tool access + independent permissions

### 4. 컨텍스트 운영 능력이 진짜 경쟁력

진화 순서: 프롬프트 엔지니어링 → 컨텍스트 엔지니어링 → [[하네스 엔지니어링]] → 에이전틱 엔지니어링

- **지속 규칙** (CLAUDE.md): 짧게 핵심만
- **작업 상태**: session state로만 유지
- **specialist 이전**: 전체 대화 아닌 요약된 작업 카드만

> 컨텍스트 운영은 기억력이 아니라 **압축력**의 문제

### 5. 짜증 신호 감지 (Frustration Detector)

```javascript
const NEGATIVE_REGEX = /wtf|wth|ffs|omfg|so frustrating|this sucks|damn it/i

function processUserPrompt(prompt) {
  const isNegative = NEGATIVE_REGEX.test(prompt.toLowerCase())
  analytics.track("tengu_input_prompt", { is_negative: isNegative })
}
```

- regex 기반 (빠르고 저렴한 규칙 엔진)
- 활용처: 위험 작업 일시 중지, UI 단순화, human handoff 후보, 운영 지표

## 완성도 높은 에이전트의 공통 구조

```
오케스트레이터 → 다음 행동 결정
specialist     → 좁은 문제 해결
tool layer     → 행동 실행
guardrail      → 차단과 승인
state          → 작업 이어 붙이기
trace          → 사후 설명
```

> Agent의 차이는 "얼마나 똑똑한가"보다 "어떻게 나누고, 잇고, 통제하느냐"에서 갈린다.
