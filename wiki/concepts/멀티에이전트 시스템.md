## 정의

멀티에이전트 시스템은 **단일 만능 에이전트** 대신 각자 명확한 역할, 권한, 컨텍스트를 가진 복수의 전문 에이전트들이 협력하는 구조다.

## 왜 역할을 나누는가

단일 에이전트에 모든 것을 맡기면:
- 왜 위험하다고 판단했는지
- 왜 그 수정이 나왔는지
- 누가 쓰기 권한을 가졌는지
- 실패 시 어디서 멈춰야 하는지

...가 모두 뒤섞여 운영 단계에서 원래 목표가 흐려진다.

## 역할 분리 패턴

```
manager → 오케스트레이션 (전체 흐름 조율)
  ↓
retrieval specialist → 읽기 전용, 근거 ID + 문서 ID + 신뢰도만 반환
  ↓
answering specialist → 근거 없으면 추가 탐색 요청
  ↓
mutation agent → manager 승인 없이는 실행 불가
```

## 역할 분리의 핵심

> "팀처럼 나눠라"가 아니라, **누가 무엇을 읽고 무엇을 쓸 수 있는지를 분명히 정하라.**

각 에이전트는 독립적으로:
- 자기 context window
- 별도 system prompt
- 특정 tool access
- independent permissions

## 실제 사례: Autopus-ADK 16개 에이전트 팀

| 에이전트 | 역할 | 권한 |
|---------|------|------|
| Planner | SPEC 분해, 태스크 배정 | 읽기 |
| Executor | TDD 구현 (병렬 worktree) | 쓰기 (격리됨) |
| Tester | 테스트 스캐폴드 + 커버리지 | 쓰기 (테스트만) |
| Reviewer | TRUST 5 코드 리뷰 | 읽기 |
| Security Auditor | OWASP Top 10 감사 | 읽기 |
| Annotator | @AX 태그 관리 | 쓰기 (주석만) |

## 모순 방지 메커니즘

에이전트 간 충돌 방지:
- 파일 소유권으로 충돌 방지
- 병렬 worktree로 격리
- 순서 기반 병합 (T1 → T2 → T3 순)
